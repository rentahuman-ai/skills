# Infrastructure Housekeeping

Enforce conventions across infrastructure-as-code, Pulumi stacks, and secret management.

## Scope

Infrastructure is distributed across two layers:

**Core IaC:**

- `packages/infra/` — Pulumi stacks: admin, ci, clickhouse, cluster, core, dns, iam, integrations, e2e-test

**Per-app IaC:**

- `apps/api/infra/` — API deployment + secrets
- `apps/rpc/infra/` — RPC service deployment + secrets
- `apps/worker/infra/` — worker deployment + secrets
- `apps/actors/infra/` — actors deployment + secrets
- `apps/electric-proxy/infra/` — electric proxy deployment + secrets
- `apps/app/infra/` — dashboard deployment
- `apps/web/infra/` — marketing site deployment

**Secrets (co-located with their stacks):**

- `packages/infra/core/secrets/` — KMS-based, per-service subdirs
- `packages/infra/integrations/*/secrets/` — integration secrets
- `apps/*/infra/secrets/` — per-app per-environment secrets

## Source of Truth

CLAUDE.md type safety rules. The dedicated `domain-infrastructure` skill is the source of truth.

## What to Check

| Check                      | Details                                                                                |
| -------------------------- | -------------------------------------------------------------------------------------- |
| SOPS rule coverage         | Every `.enc.yaml` file has a matching encryption rule in its nearest `.sops.yaml`      |
| Encryption integrity       | All `.enc.yaml` files are actually encrypted (SOPS metadata present, not plaintext)    |
| SOPS strategy consistency  | Flag mixed age/KMS strategies across dirs — is this intentional?                       |
| Auto-gen consistency       | `.sops.yaml` files generated by `generate-sops-config.ts` match current state          |
| KMS key validity           | KMS ARNs in `.sops.yaml` rules are consistent across environments                      |
| Environment naming         | Environment names consistent across Pulumi stacks and SOPS rules (staging, production) |
| Secret directory structure | Each service secret dir has its own `.sops.yaml`                                       |
| No plaintext secrets       | No unencrypted secret values in any committed file                                     |
| Pulumi type safety         | No `as`, `as any` in Pulumi stack files                                                |
| Pulumi config consistency  | All stacks have matching staging/production configs                                    |
| Script health              | `generate-sops-config.ts` scripts run without errors                                   |
| Provider freshness         | Pulumi provider versions not severely outdated                                         |

## Discovery

```bash
# All infrastructure directories
find packages/infra -name "*.ts" -not -path "*/node_modules/*" | sort
find apps/*/infra -name "*.ts" -not -path "*/node_modules/*" 2>/dev/null | sort

# SOPS configs
find . -name ".sops.yaml" -not -path "*/node_modules/*" | sort

# Encrypted files
find . -name "*.enc.yaml" -not -path "*/node_modules/*" | sort

# Pulumi configs
find . -name "Pulumi.*.yaml" -not -path "*/node_modules/*" | sort
```

## Audit Approach

Spawn 3 parallel Opus sub-agents:

**Agent 1 — Core IaC**: Read all files in `packages/infra/`. Check Pulumi type safety, config consistency, provider versions.

**Agent 2 — Secrets**: Map all `.sops.yaml` files and `.enc.yaml` files across the entire repo. Verify every encrypted file has a matching rule. Check encryption integrity. Flag strategy inconsistencies (age vs KMS).

**Agent 3 — Per-app IaC**: Read all `apps/*/infra/` directories. Check Pulumi config consistency, secrets coverage, environment naming.
